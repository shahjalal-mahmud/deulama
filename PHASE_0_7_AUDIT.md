## 4. Phase 4 — Engagement + Sync

Plan reference: §"Phase 4 — Engagement". API reference: `api.md` §"Favorites",
"Watch later", "Watched".

- [x] ✅ Verified — Favorites: `POST /api/favorites` + `DELETE /api/favorites/{id}`.
  `data/remote/api/FavoritesApi.kt` declares both endpoints with `@Body EngagementRequestDto`
  (`{ drama_id }`) and `@Path("id")`.

- [x] ✅ Verified — Watch later: `POST /api/watch-later` + `DELETE /api/watch-later/{id}`.
  Same shape. `WatchLaterApi.kt`.

- [x] ✅ Verified — Watched: `POST /api/watched` with `{ drama_id }`; **no** delete endpoint
  per api.md. `WatchedApi.markWatched(...)` only.

- [x] ✅ Verified — Optimistic local writes precede network call; reconcile on success;
  rollback on non-409 failure. `EngagementRepositoryImpls.kt` uses
  `localDao.upsert(...)` then `safeApiCall(...)` and on `Error` calls `localDao.delete(...)`.

- [x] ✅ Verified — `409 Conflict` (already applied) is treated as soft-success.
  `EngagementRepositoryImpls.kt:88-95, 124-131, 198-205` — when `safeApiCall` returns
  `Error(message.contains("409"))` they return `ApiResult.Success(Unit)`.

- [x] ✅ Verified — Anonymous writes queue to Room only.
  `EngagementRepositoryImpls.kt:85-104, 115-129, ...` short-circuit on
  `sessionManager.currentSession() == null`.

- [x] ✅ Verified — Flow-based `isFavorited` / `isQueued` / `isMarkedWatched` mirrors Room.
  `FavoritesRepository.isFavorited` etc. return `localDao.isFavoritedFlow(dramaId): Flow<Boolean>`
  implemented in each `EngagementRepositoryImpls.kt`.

- [x] ✅ Verified — Sync-on-login replay.
  `data/repository/EngagementSyncService.kt:52-90` collects `sessionManager.sessionFlow`,
  detects a fresh non-null session, and replays every queued row through the matching
  repository before clearing it.

- [x] ✅ Verified — Snackbar messages on failure use `messageFor(result)` helper.
  `EngagementRepositoryImpls.kt:276-282` maps each `ApiResult` variant to a user-facing string.

- [ ] ❌ **Bug (privacy, HIGH)** — `EngagementRepositoryImpls.listFavorites/WatchLater/Watched`
  leak **another user's** rows when the active session belongs to user A but the Room
  cache still contains user B's writes from a previous app session.
  Where: `EngagementRepositoryImpls.kt:listFavorites` (~line 264-272) reads the network
  response only — fine — BUT the **engagement state flows** (`isFavorited(dramaId)`,
  `isQueued(dramaId)`, `isMarkedWatched(dramaId)`) consult `localDao` **without
  filtering by user**. After user-A logs out and user-B logs in on the same device,
  B's DramaDetails will show A's favorited dramas as "favorited" until B saves her own.
  Fix: store `userId` on every `LocalFavoriteEntity`/`LocalWatchLaterEntity`/`LocalWatchedEntity`
  and add `WHERE userId = :currentUserId` to every read query. Repopulate or clear the
  local table inside `EngagementSyncService` when the session changes.

- [ ] ❌ **Bug (medium)** — Failure snackbars silently fall through to `"Couldn't save that
  action."` when the server returns `Error` with a blank `message` field.
  Where: `EngagementRepositoryImpls.kt:280`.
  Fix: ensure every `ApiResult.Error` produced by `safeApiCall` populates `message`
  with the raw `errorBody().string()` if the parsed `message` is empty. (Verify in
  `data/remote/ApiResult.kt:safeApiCall` — the `Error(message = ...)` branch.)

- [ ] ❌ **Bug** — `EngagementSyncService` clears the local queue **per kind** (e.g. swipeDao
  clear, favoriteDao clear) but does not invalidate Room flows after deletion, so any
  detail screen still showing "optimistically favorited" can flicker between states on
  the next composition. Investigate `EngagementSyncService.flush*` and add an explicit
  `localDao.invalidate()` or `emit(emptyList())` so `StateFlow` re-emits.

- [ ] ⚠️ Needs manual test — Favorite, kill app, re-open Drama Details; the heart icon
  should still be filled (Room-backed).

- [ ] ⚠️ Needs manual test — Disabling network, favoriting 3 dramas, re-enabling network
  and triggering sync (e.g. pull-to-refresh) replays all 3 successfully.

- [ ] ⚠️ Needs manual test — Watch-later has `409`-as-success: mark a drama as queued
  twice rapidly from two devices; expect ONE row on the server.

- [ ] ⚠️ Needs manual test — `Watched` cannot be undone from the UI (intentional, per
  spec) — confirm there's no "Unwatch" affordance anywhere.

  `data/util/ImageCompressor.kt` does the two-pass decode (inJustDecodeBounds →

- [x] ✅ Verified — `updated_fields` banner reflects what changed.
  `EditProfileViewModel.formatUpdatedFieldsBanner(updatedFields)` reads the
  server's `updated_fields` list and renders e.g. "Saved (name, password).".

- [x] ✅ Verified — Successful update writes back to `SessionManager` so the avatar /
  name appear immediately everywhere.
  `data/repository/ProfileRepositoryImpl.kt:155-164` calls
  `sessionManager.saveSession(Session(user = mergedUser, token = current.token))`.

- [x] ✅ Verified — `default.png` is never null; no null-fallback path.
  `ProfileResponseDto.image` is non-nullable; mapper drops the question by construction.

- [x] ✅ Verified — Image picker uses `ActivityResultContracts.PickVisualMedia()`
  (Android 13+ photo picker), not the legacy `GetContent`.
  `presentation/profile/EditProfileScreen.kt` `rememberLauncherForActivityResult`.

- [ ] ❌ **Bug (priority B)** — Partial-password error placement: when the user fills
  only `currentPassword` and submits, the per-field error is shown on **all three**
  password fields ("All three password fields are required together.") — but
  `currentPasswordError` reads "Current password is required." first because the field
  is non-empty. The end result is misleading: it looks like only the new-password
  fields are wrong.
  Where: `presentation/profile/EditProfileViewModel.kt:174-194` `validate()`.
  Fix: short-circuit before the per-field validation runs — if touched but not
  allThreeFilled, set the same canonical error on all three `*Error` fields and
  **return without running `validateNewPassword`/`validateConfirmPassword`**.

- [ ] ❌ **Bug (priority A, HIGH)** — `imageName` is hardcoded to `"avatar.jpg"` in
  `EditProfileViewModel.kt:114`, even for PNG / WebP uploads.
  Where: `EditProfileViewModel.kt:114` and `data/repository/ProfileRepositoryImpl.kt:73`.
  Symptom: server receives `filename="avatar.jpg"` with PNG bytes; some servers
  infer content type from filename and reject the upload as a `mime/jpeg` violation.
  Fix: derive `imageName` from the picker result. Use
  `OpenableColumns.DISPLAY_NAME` via `contentResolver.query(uri, arrayOf(DISPLAY_NAME), ...)`,
  pass it through `formState`, and forward to the multipart builder.

- [ ] ❌ **Bug (priority B)** — `ProfileRepositoryImpl` writes back to SessionManager but
  preserves the **`old`** email, even when the server includes a (potentially
  case-corrected) new email in the response. Minor today, but the plan requires the
  session to be in sync.
  Where: `data/repository/ProfileRepositoryImpl.kt:155-164`.
  Fix: build the merged `User` from the response DTO and only fall back to the cached
  email if the DTO omits it.

- [ ] ❌ **Bug** — `EditProfileScreen` does not handle the
  "image picked but the picker returned no bytes" case (e.g. URI permission denied).
  Where: `EditProfileScreen.kt:onPickImage(uri -> ...)` branch.
  Fix: in `EditProfileViewModel.onImagePicked(uri)`, call `ImageValidator.validate(...)`
  first; if it returns `failure`, populate `imageError` and don't store anything.

- [ ] ❌ **Bug** — When the user taps "Discard", the form state resets but the
  AsyncImage cache key for the previously picked image is not invalidated, so the
  picker thumb briefly lingers.
  Where: `presentation/profile/EditProfileScreen.kt:Discard button` → VM `discard()`.
  Fix: append `?v=${System.currentTimeMillis()}` to the AsyncImage `model` query so
  Coil refetches.

- [ ] ❌ **Bug** — Client-side `ImageCompressor` decodes the source URI twice
  (once for bounds, once for pixels). On pickers that return a content URI without
  `INTERNET`-readable bytes (e.g. some OEM gallery implementations), the second
  decode returns `null` and the upload silently proceeds with a 0-byte file.
  Where: `data/util/ImageCompressor.kt:compress(...)`.
  Fix: check the second decode for `null` and surface

## 9. Blocking Issues

Ordered by how likely a real user is to hit them on first run. Items here are
all ❌ — bugs that should be fixed before shipping Phase 8 polish.

1. **Phase 6 — `imageName` hardcoded to `"avatar.jpg"` (priority A)**
   `EditProfileViewModel.kt:114`, `ProfileRepositoryImpl.kt:73`. Symptom:
   any PNG / WebP upload may be rejected by filename-based MIME inference on
   strict servers. Visible to every user who tries to update their avatar.

2. **Phase 4 — Room state bleeds across accounts (privacy, HIGH)**
   `EngagementRepositoryImpls.kt` (flow queries not filtered by userId). Symptom:
   user B inherits user A's locally-cached favorites on the same device until A's
   queue is overwritten by B's first write. Worst case: B sees A's favorited
   dramas marked "favorited" on DramaDetails.

3. **Cross-cutting — `fallbackToDestructiveMigration()`**
   `data/local/db/DatabaseModule.kt:30`. Any schema bump in a release wipes every
   user's anonymous swipe / favorite / watch-later / watched queue with zero
   warning. With Phase 4 sync-on-login the cost is bounded, but it's still a
   data-loss vector for anonymous users.

4. **Phase 1 — No "session expired" snackbar**
   `presentation/navigation/NavGraph.kt:88-95`. Symptom: silent logout on
   token expiry; user is dumped on Login with no explanation.

5. **Phase 7 — `load()` race on rapid retries**
   `presentation/activity/ActivityViewModel.kt:54-77`. Symptom: a slow first
   load can overwrite a faster second load's `Success` with `Error` if the
   user taps Retry between them.

6. **Phase 3 — Swipe deck has no rollback on network failure**
   `presentation/swipe/SwipeViewModel.kt` + `SwipeRepositoryImpl.kt`. Symptom:
   swipe disappears visually but server never records the action. After enough
   offline swipes the dataset is silently inconsistent.

7. **Phase 3 — Anonymous swipes don't queue to Room**
   `SwipeRepositoryImpl.kt:18-21`. Symptom: anonymous swipe tab feels broken;
   nothing is recorded for sync-on-login.

8. **Phase 6 — Partial-password error placement is misleading (priority B)**
   `EditProfileViewModel.kt:174-194`. Symptom: filling only `current_password`
   shows the wrong field as the offender.

9. **Phase 2 — No search / sort / genre on Discover**
   `presentation/discover/DiscoverScreen.kt`, `DiscoverViewModel.kt:23`.
   Symptom: spec'd filter UI is entirely missing; users scroll the entire
   `created_at` paginated catalog with no recourse.

10. **Phase 5 — Recommendations stale on tab re-entry**
    `presentation/recommendations/RecommendationsViewModel.kt`. Symptom: feed
    doesn't update between sessions; user has to force-kill the app.

11. **Phase 5 — No SignedOut CTA on Recommendations / Genre Stats**
    Both screens. Symptom: a 401 lands on a generic error banner with no path
    back to login.

12. **Cross-cutting — Zero `stringResource` use**
    `app/src/main/java/**` (grep). Symptom: every string is burned-in English.
    Prevents shipping in Bengali (the app's name suggests target locale) or
    supporting RTL.

13. **Phase 6 — Image pick null-bytes silently uploaded**
    `data/util/ImageCompressor.kt:compress(...)`. Symptom: vague 422 from
    server after the picker hands back a URI the app can't actually read.

14. **Phase 4 — Failure snackbars fall through to blank "Couldn't save that
    action."**
    `EngagementRepositoryImpls.kt:280` + `data/remote/ApiResult.kt`. Symptom:
    users get an unhelpful toast on real failures.

15. **Phase 6 — `Discard` doesn't invalidate Coil cache key**
    `presentation/profile/EditProfileScreen.kt`. Symptom: stale picked-image
    thumb lingers for a frame after discard.

16. **Phase 5 — Cold-start banner copy is duplicated across screens with
    minor drift.**
    `RecommendationsScreen.kt` + `GenreStatsScreen.kt`.

17. **Phase 6 — `ProfileRepositoryImpl` preserves old email even when the
    response includes a (case-corrected) new one.**
    `ProfileRepositoryImpl.kt:155-164`. Minor.

18. **Cross-cutting — `MappersPlaceholder.kt` left in the codebase** (verify
    with Glob). Cleanup.
